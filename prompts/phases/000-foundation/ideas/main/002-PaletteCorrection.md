# 002-PaletteCorrection

## 1. 背景 (Background)
`monitor` モジュールの 8bit パレットから RGB への計算ロジック（`r := byte(((p >> 5) & 7) * 255 / 7)`等）において、Goの `byte` (uint8) 型で計算が行われているため、乗算 `* 255` の時点でオーバーフローが発生し意図しない暗い色になっています。
例えば `7 * 255` は 1785 ですが、8bitのオーバーフローによって `249` に切り詰められ、その後に `/ 7` されるため、最大値が `35` という非常に暗い値に制限されてしまっています。この結果、画面上に表示されるピクセルが本来の明るさ（白なら255）になりません。

## 2. 要件 (Requirements)

**必須要件**:
1. **パレット計算式のオーバーフロー修正**: Monitor モジュールの VRAM受信ループ内にある RGB マッピング計算で、`uint8` のまま乗算しないように計算用の型を適切に拡張（`int` など）してから割り算し、再度 `byte` に戻す処理に修正する。
2. **256色（8-bit）カラー空間への完全な展開**:
   - `x=255` に該当する 256個目のカラー（ビット配列 `11111111`）が正しく (255, 255, 255) の白として画面出力されること。

## 3. 実現方針 (Implementation Approach)

*   **型キャストの追加**:
    `features/vm/internal/modules/monitor/monitor.go` の `receiveLoop` における計算式を以下のように修正します。
    ```go
    // p は byte型
    r := byte((int((p >> 5) & 7) * 255) / 7)
    g := byte((int((p >> 2) & 7) * 255) / 7)
    b := byte((int(p & 3) * 255) / 3)
    ```
    これにより、乗算が `int` 型空間で行われるためオーバーフローしなくなります。

## 4. 検証シナリオ (Verification Scenarios)

(1) 開発環境にて `cd features/vm` を実行後、`go run ./cmd/main.go` を実行する。
(2) 表示されたモニターウィンドウ上部に描画される横線の右側のピクセルカラー（とくに `x=255` に近い部分）が、暗い色ではなく、はっきりと真っ白になっていることを目視確認する。
