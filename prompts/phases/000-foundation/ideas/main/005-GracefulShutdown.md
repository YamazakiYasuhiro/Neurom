# モジュール・グレースフルシャットダウン仕様書

## 1. 背景 (Background)

### 1.1. 課題
現在のアーキテクチャでは各モジュール（CPU、VRAM、Monitorなど）が独立したプロセス・Goroutineとして起動されますが、システム全体を一斉かつ安全に終了させるための仕組み（グレースフルシャットダウン）が不足しています。
ユーザーが手動でMonitorウィンドウの「閉じる」ボタンを押した際、ウィンドウだけが閉じられてしまい、背後で動いているGoroutineやZeroMQ等のリソースが適切に解放されずにプロセスが残留してしまう懸念があります。システムの一貫したライフサイクル管理のために、この問題を解決する必要があります。

## 2. 要件 (Requirements)

### 2.1. 各モジュールのグレースフルシャットダウン
- 全てのモジュールは、システムバス（ZeroMQ）を介して特定のコマンド（シャットダウンメッセージ）を受信した場合に、イベントループを安全に終了すること。
- 必要に応じて確保しているリソース（ソケットのクローズ等）をクリーンアップし、親プロセス側に対して処理が終了したことを返すこと。

### 2.2. Monitorウィンドウからの終了メッセージ送信
- Monitorモジュールのウィンドウにある「閉じる」ボタン（または等価なウィンドウクローズイベント）を押した際のアクションをフックすること。
- ウィンドウが完全に破棄される直前に、Monitorモジュール自身が「シャットダウンメッセージ」をZeroMQのシステムバスに Publish （送信）し、他の全てのモジュールへシステム終了を通知すること。

### 2.3. システム終了制御の共通化
- 各モジュール間の抽象コマンドメッセージとして「システム終了」を意味する一意の識別子またはフォーマットを定義すること。

## 3. 実現方針 (Implementation Approach)

### 3.1. シャットダウンメッセージの定義
- 既存のコマンドメッセージ体系に、対象(Target)が `System`、操作(Operation)が `Shutdown` となるメッセージ等を定義します。

### 3.2. モジュールの終了ハンドリング
- パブリッシャーやサブスクライバーにおける無限ループ内（`for` と `Select` / `.Recv` 等の箇所）で、System Shutdownコマンドを検知した際にループを `break` して関数から復帰するように実装します。
- `ModuleManager`（またはメイン関数）等は、各Goroutineの終了を `sync.WaitGroup` 等を用いて待ち合わせ、全てが完了した段階でメインプロセス自体を終了させます。

### 3.3. MonitorモジュールへのPublisher機能追加
- 既存の仕様ではMonitorを含め、いくつかのはSubscriberとしての機能しか持っていませんでしたが、終了時にメッセージを投げるため、Monitorモジュールにも（少なくともバス向けの）Publisherの実装または同等の送信手段を導入します。
- GUIウィンドウ枠の閉じるイベント（`destroy` またはそれに類するLifecycleイベント等）をポーリングまたはコールバックで検知し、メッセージを送信します。

### 3.4. Windows環境におけるx/mobileの注意事項
- Windows上の `x/mobile` は、ウィンドウが `lifecycle.StageDead` に到達した後も、ウィンドウを再作成しようとして `StageDead → StageVisible` のイベントを発行し続ける。そのため `a.Events()` チャネルは閉じられず、`for e := range a.Events()` ループは自然には終了しない。
- この問題を回避するため、`StageDead` を検知した時点でラベル付き `break` によりイベントループを強制的に脱出する必要がある。

## 4. 検証シナリオ (Verification Scenarios)

1. エミュレータ本体（CPU、VRAM、Monitor等を含む全モジュール）を通常通り起動する。
2. モニタ（GUIウィンドウ）が正常に表示されることを確認する。
3. モニタウィンドウのシステム標準の「閉じる」ボタン（×ボタン等）をクリックする。
4. Monitorモジュール内部のイベントフックからシャットダウンメッセージ(`SYSTEM_SHUTDOWN`等)がPublishされる。
5. CPU、VRAM、およびMonitor自身のバックグラウンドタスクがバス経由でシャットダウン要求を受信し、内部ループを抜け、確保しているリソースを解放して終了する。
6. プロセスマネージャ（メインスレッド）が各モジュールの終了を検知し、システム全体例外やエラーを吐くことなく終了コード0でOSに制御を戻す（コンソールプロセスが消滅する）ことを確認する。
