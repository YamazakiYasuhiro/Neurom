# 007 - ZMQ Bus Panic Guard

## 背景

Neurom 起動時、Monitor モジュールのライフサイクルイベント直後にプロセスが panic する:

```
panic: runtime error: makeslice: len out of range

goroutine 6 [running]:
github.com/go-zeromq/zmq4.(*Conn).read(0xc000110320)
        .../zmq4@v0.17.0/conn.go:416 +0x1af
github.com/go-zeromq/zmq4.(*qreader).listen(...)
        .../zmq4@v0.17.0/msgio.go:105 +0xae
```

### 原因分析

1. **クラッシュ箇所**: `zmq4/conn.go:416` — `body := make([]byte, size)` で、`size` は ZMTP フレームヘッダから `uint64` として読み取られる。
2. **不十分なガード**: zmq4 内部のガードは `size > uint64(maxInt64)` (~9.2 EB) のみ。これより小さいが物理メモリを大幅に超える値はチェックをすり抜け、Go ランタイムが `makeslice: len out of range` で panic する。
3. **回復不能な goroutine**: panic は zmq4 が内部的に起動する `qreader.listen` goroutine で発生する。この goroutine には `recover()` がなく、アプリケーションコードからアクセスもできない。
4. **発生条件**: `NewZMQBus` が PUB/SUB ソケットを `inproc://system-bus` と `tcp://127.0.0.1:5555` の **両方** にバインド/接続している。TCP エンドポイントは以下の要因に脆弱:
   - 起動時の ZMTP ハンドシェイクのタイミング問題
   - 前回クラッシュ時の残留データによるポート再利用
   - 外部プロセスがポート 5555 に接続し非 ZMTP データを送信
   - ネットワークレベルのデータ破損

### 大規模データ転送の課題

現在の `BusMessage` は単一の `Data []byte` フィールドでペイロード全体を送信する。現時点で最大のメッセージは `blit_rect`（全画面 256×212 = ~54 KB）で問題ないサイズだが、以下の理由からチャンク送信の仕組みが必要:

- **解像度拡大への対応**: 将来的に解像度が上がればデータ転送量は比例して増加する（例: 512×424 で ~218 KB、1024×848 で ~870 KB）
- **ZMQ フレーム破損リスクの低減**: フレームが小さいほど、破損時の影響範囲が限定される
- **TCP ブリッジとの親和性**: 小さいフレームは転送・再送が容易
- **VRAM モジュールの既存ワークアラウンド解消**: 現在 `maxCompactEventPixels = 4096` でデータを打ち切る回避策がある。チャンク送信があればこの制限を撤廃できる

### ライブラリの状況

- `go-zeromq/zmq4` v0.17.0 が最新リリース（2024年5月）。
- ライブラリは `[WIP]` 表記で、メンテナは時間的制約を明示している。
- フレームサイズ上限を設定する `OptionMaxMsgSize` 等のソケットオプションは存在しない。
- 上流での修正は利用不可であり、近い将来の修正も見込めない。

> **[対応済み]** 実装調査により、zmq4 v0.17.0 の inproc トランスポート (`internal/inproc/conn.go`) に追加のバグを発見。`conn.Read()` が Write より小さいバッファで呼ばれた場合、余剰データを破棄し ZMTP ストリームの同期を失う。255 バイトを超える ZMTP ロングフレームで必ず発生するため、内部バスを Go チャネルベースの `ChannelBus` に置き換え、zmq4 は TCP ブリッジ専用に限定した。

## 要件

### 必須

1. **R1: 内部通信と外部通信のトランスポート分離** — 内部モジュールバス（CPU, VRAM, IO, Monitor が使用する `Bus` インターフェース）は `inproc://` トランスポートのみを使用しなければならない。TCP エンドポイントを内部モジュール通信に使用してはならない。
2. **R2: TCP エンドポイントの隔離** — TCP が外部接続（デバッグ、ツール連携）に必要な場合、独自の panic recovery を備えた分離されたブリッジプロセスまたは goroutine として実行しなければならない。TCP 側の panic がエミュレータ全体を巻き込まないこと。
3. **R3: バス起動時に panic しないこと** — Neurom の起動（`./bin/neurom.exe`）は、高速再起動、ポート再利用、外部接続なしを含むあらゆる通常の動作条件下で panic してはならない。
4. **R4: TCP の graceful degradation** — TCP ブリッジが致命的エラー（回復された panic を含む）に遭遇した場合、エラーをログに記録し、TCP ブリッジのみを再試行またはシャットダウンしなければならない。inproc ベースのモジュール通信に影響を与えてはならない。
5. **R5: 後方互換性** — `Bus` インターフェース（`Publish`, `Subscribe`, `Close`）は変更してはならない。既存モジュールのコード変更を必要としてはならない。
6. **R6: バス層でのチャンク送信** — `ZMQBus` の `Publish` 内部で、エンコード後のペイロードが閾値（例: 32 KB）を超える場合、自動的に複数チャンクに分割して送信しなければならない。`receiveLoop` 内部でチャンクを再組み立てし、完全な `BusMessage` としてサブスクライバに配信する。モジュール側はチャンクの存在を意識しない（完全な透過性）。

### 任意

7. **R7: 最大フレームサイズのセーフガード** — 上流の修正またはラッパーで ZMTP フレームサイズを妥当な値（例: 64 MB）に制限できる場合、追加の防御レイヤーとして実装する。
8. **R8: TCP ブリッジ無効化フラグ** — TCP エンドポイントが不要な環境向けに、CLI フラグ（例: `--no-tcp`）を追加し、TCP エンドポイントを完全に無効化可能にする。

## 実現方針

### アーキテクチャ概要

```
変更前（現状 — デュアルトランスポートの単一 ZMQBus）:
┌───────────────────────────────────────────┐
│ ZMQBus                                    │
│  PUB: Listen(inproc + tcp)                │
│  SUB: Dial(inproc + tcp)                  │
│  receiveLoop → channels                   │
└───────────────────────────────────────────┘

変更後（提案 — inproc 専用バス + チャンク + オプショナル TCP ブリッジ）:
┌──────────────────────────────┐     ┌─────────────────────────┐
│ ZMQBus (inproc のみ)         │     │ TCPBridge (オプション)    │
│  PUB: Listen(inproc)         │◄───►│  PUB: Listen(tcp)        │
│  SUB: Dial(inproc)           │     │  SUB: Dial(inproc)       │
│  Publish: 自動チャンク分割    │     │  recover() で保護         │
│  receiveLoop: チャンク再組立  │     │  障害時に再起動           │
│  → 完全なメッセージを配信     │     └─────────────────────────┘
└──────────────────────────────┘
```

### 主要な変更点

1. **`cmd/main.go`**: `NewZMQBus` に `inproc://` エンドポイントのみを渡す。TCP エンドポイントは、`inproc` と `tcp` 間のメッセージを中継する別の `TCPBridge` コンポーネントが処理する。
2. **`bus/tcpbridge.go`（新規）**: TCP ブリッジを実装:
   - 独自の PUB ソケット（TCP）と SUB ソケット（inproc）
   - `recover()` 付きの goroutine で panic 時に再起動
   - 最大再起動回数の設定
   - panic イベントのロギング
3. **`bus/zmqbus.go`**: チャンク送信・再組み立てロジックを追加:
   - 構造的な変更なし（`Bus` インターフェースは不変）
   - `inproc://` エンドポイントのみを受け取る
4. **CLI フラグ**: `--no-tcp` フラグを追加し、TCP ブリッジの初期化を完全にスキップ可能にする。

### チャンク送信の設計

#### ワイヤフォーマット

チャンクは ZMQ マルチフレームメッセージのペイロード部分（トピックフレームの次のフレーム）に適用する。

```
通常メッセージ（閾値以下）:
  Frame[0]: topic (既存のまま)
  Frame[1]: gob-encoded BusMessage (既存のまま)

チャンクメッセージ（閾値超過）:
  Frame[0]: topic (既存のまま)
  Frame[1]: chunk header + chunk payload

チャンクヘッダ (固定 17 bytes):
  [0]      : マジックバイト 0xC1 (チャンクメッセージの識別)
  [1..8]   : メッセージ ID (uint64, big-endian) — 同一メッセージのチャンクを紐付ける
  [9..10]  : チャンクインデックス (uint16, big-endian) — 0 始まり
  [11..12] : 総チャンク数 (uint16, big-endian)
  [13..16] : 元メッセージの総バイト数 (uint32, big-endian)
```

#### 送信側 (`Publish`)

```
1. BusMessage を gob エンコードする（既存処理）
2. エンコード後のサイズが ChunkThreshold（デフォルト 32 KB）以下なら従来通り単一送信
3. 超過する場合:
   a. メッセージ ID を atomic カウンタから採番
   b. ペイロードを ChunkThreshold ごとに分割
   c. 各チャンクにヘッダを付与して順次 Send
```

#### 受信側 (`receiveLoop`)

```
1. 受信したフレームの先頭バイトを確認
2. マジックバイト 0xC1 でなければ従来通り gob デコード → 配信
3. マジックバイト 0xC1 なら:
   a. チャンクヘッダを解析
   b. メッセージ ID をキーに reassembly バッファに格納
   c. 全チャンクが揃ったら結合 → gob デコード → 配信
   d. タイムアウト（例: 5秒）を超えた不完全なバッファは破棄
```

#### 設計上の判断

- **マジックバイト `0xC1`**: gob エンコードの先頭バイト（通常 `0x0`〜`0x1F` の型ヘッダ）と衝突しないため、チャンクメッセージを確実に識別できる。
- **uint16 のチャンクインデックス/総数**: 最大 65,535 チャンク × 32 KB = ~2 GB をカバー。将来的な拡張にも十分。
- **reassembly タイムアウト**: チャンクが欠落した場合にメモリリークを防ぐ。
- **送信順序の保証**: ZMQ PUB/SUB の同一接続上では送信順序が保持されるため、チャンクの順序逆転は発生しない。ただし堅牢性のため、インデックスベースの組み立てを行う。

### この方針が有効な理由

- **inproc トランスポート** はプロセス内メモリバッファを使用する。TCP スタック、ネットワークソケット上のバイトレベルフレーミング、外部データへの露出がない。ZMTP のハンドシェイクとデータ交換は同一プロセスのメモリ空間内で行われるため、フレームヘッダ破損を引き起こすバグのクラスが排除される。
- **TCP ブリッジの隔離** により、TCP 側で panic が発生しても（不正な外部データなど）、`recover()` がブリッジ goroutine 内でキャッチし、ログに記録して必要に応じて再起動する。エミュレータ本体はクラッシュしない。
- **チャンク送信のバス層隠蔽** により、モジュール側は任意サイズのデータを `Publish` するだけでよく、分割・再組み立ての詳細を知る必要がない。VRAM モジュールの `maxCompactEventPixels` のようなサイズ制限ワークアラウンドも将来的に撤廃可能。

## 検証シナリオ

1. Neurom の通常起動（`./bin/neurom.exe`）:
   - (1) バイナリを実行する
   - (2) "All modules started successfully." が出力されることを確認
   - (3) Monitor ウィンドウが panic なしで開くことを確認
   - (4) ウィンドウを正常に閉じる
   - (5) "Shutdown complete. Exiting process." が出力されることを確認

2. ヘッドレスモードでの起動（`./bin/neurom.exe --headless`）:
   - (1) `--headless` で実行する
   - (2) バス経由でシャットダウンコマンドを送信する
   - (3) プロセスが panic なしでクリーンに終了することを確認

3. TCP 無効での起動（`./bin/neurom.exe --no-tcp`）:
   - (1) `--no-tcp` で実行する
   - (2) モジュールが inproc 経由で正常に通信することを確認
   - (3) TCP ポートが開かれていないことを確認

4. TCP ブリッジの panic 回復:
   - (1) TCP ブリッジ付きで Neurom を起動する
   - (2) 外部クライアントを TCP ポートに接続し、不正な（非 ZMTP）データを送信する
   - (3) TCP ブリッジがエラーをログに記録して回復することを確認
   - (4) 内部モジュール通信が影響を受けないことを確認

5. 高速再起動（ポート再利用）:
   - (1) Neurom を起動する
   - (2) プロセスを即座に Kill する（例: Ctrl+C）
   - (3) 1 秒以内に Neurom を再起動する
   - (4) panic が発生しないことを確認

6. チャンク送信の透過性:
   - (1) 閾値（32 KB）を超えるデータを持つ BusMessage を Publish する
   - (2) Subscribe 側で受信したメッセージが元のメッセージと完全に一致することを確認
   - (3) 閾値以下のメッセージも従来通り正しく配信されることを確認

7. チャンク送信の境界値:
   - (1) ちょうど閾値サイズのメッセージを送信する
   - (2) 閾値 +1 バイトのメッセージを送信する
   - (3) 空データのメッセージを送信する
   - (4) すべて正しく配信されることを確認

### 具体的なテストケース

1. **`TestZMQBus_InprocOnly`**: `inproc://` エンドポイントのみで `ZMQBus` を作成。Publish と Subscribe を行い、メッセージ配信を検証する。R1 を確認。
2. **`TestTCPBridge_PanicRecovery`**: TCP ブリッジを起動し、ブリッジ goroutine で panic をシミュレートする。ブリッジが回復し、メインバスが影響を受けないことを検証する。R2 と R4 を確認。

7. **`TestZMQBus_ChunkBoundary`**: ちょうど閾値サイズ、閾値 +1 バイト、空データのメッセージをそれぞれ送受信し、正しく処理されることを確認する。R6 の境界値テスト。
8. **`TestZMQBus_ChunkReassemblyTimeout`**: チャンクの一部だけを送信し、タイムアウト後に不完全なバッファが破棄されることを確認する。メモリリーク防止の検証。
